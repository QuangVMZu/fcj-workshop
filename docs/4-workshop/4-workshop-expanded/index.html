Workshop Expanded

Hands-on AWS Workshop for the EV Charging Platform

Overview

This expanded guide elaborates the existing 4-Workshop chapter and its seven subsections into a single production-oriented reference. It still follows the original workshop intent: explain the deployed AWS architecture behind the EV Charging Station Management System rather than local application setup. The difference is depth. Each subsection below is expanded with additional architectural context, higher-level internal logic, request and response behavior, practical assumptions, scaling guidance, security notes, and realistic improvement paths.

From an architecture perspective, the workshop describes a layered AWS platform with a clear separation between public entry points, private runtime components, and operational controls. The frontend is delivered from the edge, the backend runs on private compute, the relational database remains isolated, and observability plus administrative access are handled through AWS-native control-plane services.

How to Use This Workshop

  • Follow the sections in order from edge services to observability, because each subsection builds on the layer before it.
  • Replace placeholder values such as ${APP_DOMAIN} and ${DB_INSTANCE_ID} with your own environment values before running any commands.
  • The CLI examples use bash syntax. If you use PowerShell, adapt quoting and variable handling accordingly.
  • Prefer a non-production account for experiments such as Route 53 changes, CloudFront invalidations, or RDS failover tests.
  • Read each section in two passes: first for the flow, then for the commands and verification steps.
  • Treat each subsection as both a learning module and an operational playbook. The explanations describe the intended design, while the commands show how to validate that the environment actually matches that design.
  • When a section mentions a service that is not drawn explicitly in the diagram, treat it as a reasonable supporting assumption rather than a confirmed deployed component.

Services Used in This Architecture

  • Edge and frontend delivery: Amazon Route 53, Amazon CloudFront, AWS WAF, Amazon S3, Amazon Location Service
  • Application runtime: Amazon VPC, Internet Gateway, NAT Gateway, Application Load Balancer, Amazon EC2, EC2 Auto Scaling
  • Data layer: Amazon RDS in a primary and standby layout
  • Operations and security: AWS Systems Manager, IAM instance profile, AWS Secrets Manager, AWS KMS
  • Observability and communication: Amazon CloudWatch, AWS CloudTrail, Amazon SES, Route 53 email DNS records

These services form a practical AWS web-application baseline. Route 53 and CloudFront accept user traffic, S3 serves the static frontend, ALB and EC2 handle synchronous backend processing, RDS stores transactional state, and supporting services provide operator access, encryption, logging, audit, and outbound messaging.

Services Not Shown in This Diagram

  • API Gateway: Not used here because public application traffic enters through CloudFront and then the Application Load Balancer. In a future design, API Gateway would be appropriate only if the platform needed API products, request validation, authorizers, per-client throttling, or usage plans.
  • Lambda: Not shown. The backend is modeled as long-running EC2 instances. Lambda could still be introduced later for utility automation, asynchronous cleanup tasks, or SES event processing.
  • ECS / EKS: Not shown. Container orchestration is outside the scope of this design. The workshop intentionally stays focused on an EC2-based runtime model.
  • DynamoDB: Not shown. Persistent data is modeled as relational data in Amazon RDS. DynamoDB would make sense only for future low-latency lookup, idempotency, or event-driven read models.
  • EventBridge / SQS / SNS: Not shown. The diagram emphasizes synchronous request flow and direct service integrations. These services become relevant when retries, asynchronous notifications, or background jobs need better isolation.
  • Cognito: Not shown. The visible admin path uses IAM and AWS Systems Manager rather than a user pool.

  • Read-only access is enough for most workshop checks.

Workshop Overview and AWS Service Map

System Overview

  • The EV Charging Station Management System is deployed as a layered AWS web platform.
  • End users access a browser-based frontend through Route 53 and CloudFront.
  • Static assets are stored in Amazon S3 and protected behind CloudFront Origin Access Control.
  • Backend requests are routed to private EC2 instances through an Application Load Balancer.
  • Business data is stored in Amazon RDS and isolated from public access.
  • Operations teams use AWS-native tooling such as Systems Manager, Secrets Manager, CloudWatch, CloudTrail, and SES.
  • The architecture separates content delivery, synchronous business processing, data persistence, and operational governance so that each layer can be tuned independently.
  • The design assumes the backend is state-light at the instance level, which makes horizontal scale through Auto Scaling more realistic.

What this subsection does:

  • It establishes the shared system map used by the remaining six subsections.
  • It explains why the workshop is organized by architectural layer rather than by AWS product name.
  • It gives new readers a practical way to tell which services are public-facing, which are private, and which are operator-only.

Why it exists:

  • Teams troubleshoot faster when they understand the full service chain before looking at a single component.
  • Many AWS incidents are boundary issues rather than single-service failures, so the overview makes those boundaries explicit.
  • The subsection prevents incorrect assumptions such as “the app uses Lambda” or “the database is public” by showing the intended architecture up front.

End-to-End AWS Flow

  1. A user resolves ${APP_DOMAIN} through Amazon Route 53. Service: Amazon Route 53. What happens: Route 53 answers the public DNS query and directs the client to the CloudFront distribution. Why it matters: DNS is the first control point for the public application name, so a mistake here breaks the entire chain.
  2. CloudFront receives the request and evaluates AWS WAF rules. Services: Amazon CloudFront and AWS WAF. What happens: The edge layer terminates HTTPS, enforces edge security policy, and decides whether the request can continue. Why it matters: The edge layer reduces latency and filters malicious requests before they hit the backend.
  3. CloudFront serves cached frontend assets or fetches them from Amazon S3. Services: Amazon CloudFront and Amazon S3. What happens: Static content is delivered directly from cache when possible, otherwise fetched privately from S3. Why it matters: Static delivery stays fast and the S3 bucket does not need to be public.
  4. The frontend calls backend APIs through the Application Load Balancer. Service: Application Load Balancer. What happens: Dynamic requests are routed toward the application layer rather than the static-delivery path. Why it matters: The ALB is the public HTTP entry point for dynamic requests.
  5. The ALB routes traffic to healthy EC2 instances in private subnets. Services: ALB, Amazon EC2, EC2 Auto Scaling. What happens: Target-group health and listener logic decide which instance should process the request. Why it matters: The backend can scale horizontally without exposing instances directly.
  6. The EC2 application reads or writes relational data in Amazon RDS. Service: Amazon RDS. What happens: Transactional operations flow over private networking to the managed relational database. Why it matters: Transactions stay in a managed relational service with standby support.
  7. The application retrieves secrets, writes logs, and sends email by using its IAM role. Services: IAM, Secrets Manager, KMS, CloudWatch, SES. What happens: The workload uses temporary AWS credentials from its instance profile to reach supporting services. Why it matters: The instance does not need static credentials stored on disk.
  8. Operators administer the environment through Systems Manager and audit activities in CloudTrail. Services: AWS Systems Manager and AWS CloudTrail. What happens: Administrative activity goes through AWS-native control channels and remains auditable. Why it matters: Operations remain traceable and do not require public SSH access.

Request and response behavior:

  • User-facing browser traffic receives either cached static assets from CloudFront or dynamic JSON/API responses from the ALB-backed application tier.
  • Backend-to-database traffic is private, stateful, and transactional.
  • Control-plane and observability traffic is not part of the direct user request path, but it is essential for secure operations and post-incident analysis.

Hands-on Baseline Checks

Run these commands before working through the remaining sections.

aws sts get-caller-identity
aws route53 list-hosted-zones --region ${AWS_REGION}
aws cloudfront list-distributions --region ${AWS_REGION}
aws ec2 describe-instances --region ${AWS_REGION} --max-items 5
aws rds describe-db-instances --region ${AWS_REGION}

What to look for:

  • You can authenticate to the intended AWS account.
  • The hosted zone, CloudFront distribution, EC2 instances, and RDS instance all exist in the expected environment.
  • The region and naming convention match the values you plan to use in the later labs.
  • The discovered resource names are consistent enough that automation, dashboards, and runbooks can be built around them.

Beginner Guidance

  • Start by understanding which components are public and which are private.
  • Treat Route 53, CloudFront, ALB, and SES as public-facing services.
  • Treat EC2, RDS, Secrets Manager, and KMS as controlled internal services.
  • Treat Systems Manager, CloudWatch, and CloudTrail as operator tools.
  • If a service is not shown in the diagram, do not assume it exists in the deployment.
  • Read the architecture from left to right for request flow, and from top to bottom for trust boundaries and operational ownership.

Notes / Assumptions

  • The diagram shows infrastructure topology, not application source code or CI/CD pipelines.
  • The existing project proposal mentions Amazon RDS for SQL Server, but the diagram itself labels only Amazon RDS. Adjust engine-specific commands as needed.
  • No API Gateway, Lambda, ECS, EKS, DynamoDB, EventBridge, SQS, SNS, or Cognito components are visible in this architecture.
  • TLS certificates, security groups, subnet route tables, and EC2 launch templates are not drawn, but they are required for this design to function in a real AWS account.

Hands-on Edge Delivery and Geospatial Requests

Overview

  • This section focuses on the public edge of the platform.
  • It explains how DNS, CDN caching, WAF filtering, S3 origin access, and Location Service requests work together.
  • It is the right place to start when the frontend is slow, the domain does not resolve, or map data is missing.
  • It also defines the boundary between static web delivery and the later dynamic request path to the backend.
  • From an AWS architecture perspective, this subsection is the platform’s latency-optimization and first-line security layer.

What this step does:

  • It brings browsers into the system through Route 53 and CloudFront.
  • It protects the public ingress path with WAF and private-origin S3 access.
  • It serves map and place-search capabilities without forcing all geospatial traffic through the backend.

Why it exists:

  • Static assets and map requests have very different caching and scaling behavior from transactional API calls.
  • Frontend delivery should be globally distributed and cost-efficient.
  • The edge path should absorb common attacks before they consume application capacity.

What problem it solves:

  • Slow or inconsistent global frontend delivery
  • Unnecessary origin load on S3 or the backend
  • Public exposure of the frontend bucket
  • Poor separation between web-content delivery and geospatial integration

AWS Services in This Section

Amazon Route 53

  • Resolves the public domain and directs users to the appropriate entry point.
  • Acts as the first step in the request flow.
  • Provides a controlled place to manage subdomains, validation records, and failover-ready DNS policy if the architecture evolves.

Amazon CloudFront

  • Delivers cached frontend assets with low latency.
  • Reduces repeated requests to the origin (S3).
  • Terminates HTTPS close to the user and can enforce cache policy, header behavior, and origin controls.

AWS WAF

  • Filters incoming HTTP/HTTPS requests.
  • Protects the system from common web attacks at the edge.
  • Is the natural place to apply rate-based controls, managed rules, and coarse bot filtering before requests hit the application path.

Amazon S3

  • Stores static frontend assets (HTML, JS, CSS).
  • Serves as the origin for CloudFront via Origin Access Control.
  • Keeps frontend deployment simple by separating object delivery from application runtime.

Amazon Location Service

  • Handles geospatial requests such as map rendering and place search.
  • Operates independently from the static asset delivery flow.
  • Lets the frontend consume map capabilities directly instead of overloading the core backend with proxy-style requests.

Step-by-Step AWS Flow

  1. A user opens ${APP_DOMAIN} in the browser. Service: Amazon Route 53. What happens: Route 53 resolves the public hostname to the CloudFront distribution. Why it matters: If DNS is wrong, no later AWS service can help.
  2. CloudFront receives the HTTPS request. Services: Amazon CloudFront and AWS WAF. What happens: WAF rules inspect the request before the distribution serves content. Why it matters: Malicious or malformed traffic can be blocked at the edge.
  3. CloudFront checks whether the requested file is cached. Services: Amazon CloudFront and Amazon S3. What happens: Cached files are returned immediately; cache misses are fetched from S3. Why it matters: Cache hits reduce latency and lower load on the origin bucket.
  4. S3 returns frontend objects only through the approved CloudFront path. Services: Amazon S3 and CloudFront Origin Access Control. What happens: The bucket stays private while CloudFront remains the allowed reader. Why it matters: Users cannot bypass CloudFront and read objects directly from S3.
  5. The frontend later requests geospatial data. Service: Amazon Location Service. What happens: The browser or frontend code calls place index and map APIs. Why it matters: Map traffic is separated from static asset delivery.
  6. The frontend then calls backend APIs through the application layer. Service: Application Load Balancer in the next section. What happens: Dynamic traffic leaves the edge path and enters the private compute stack. Why it matters: Static and dynamic traffic can be scaled and tuned independently.

Request and response behavior

  • DNS response: Route 53 returns the CloudFront entry that the browser should reach.
  • Static asset response: CloudFront returns cached content or forwards an origin fetch to S3.
  • Geospatial response: Amazon Location Service returns search or map data directly to the frontend integration.
  • Dynamic handoff: Once the SPA is loaded, API calls leave the edge-delivery path and move to the ALB-based application tier.

High-level internal logic

  • CloudFront is effectively the read-optimization layer for the frontend.
  • S3 is the object store of record for frontend artifacts, but not the public endpoint.
  • WAF is the edge gatekeeper.
  • Amazon Location Service is a specialized API dependency used by the UI, not the main application server.

AWS CLI Walkthrough

1. Inspect the hosted zone

aws route53 list-hosted-zones --region ${AWS_REGION}

Why this check exists:

  • It verifies that DNS is managed in the expected account and region context.

2. Create or update the application DNS record

Create a file named route53-app-record.json:

{
  "Comment": "Point app subdomain to CloudFront",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "CNAME",
        "TTL": 300,
        "ResourceRecords": [
          {
            "Value": "d111111abcdef8.cloudfront.net"
          }
        ]
      }
    }
  ]
}

Apply the change and wait until it is in sync:

CHANGE_ID=$(aws route53 change-resource-record-sets \
  --hosted-zone-id ${HOSTED_ZONE_ID} \
  --change-batch file://route53-app-record.json \
  --query 'ChangeInfo.Id' \
  --output text)

aws route53 wait resource-record-sets-changed --id ${CHANGE_ID}

Why this check exists:

  • The UPSERT model is safe for repeated infrastructure updates.
  • Waiting for sync avoids validating the route before DNS changes have propagated inside Route 53.

3. Deploy frontend assets to S3

aws s3 sync ./dist s3://${STATIC_BUCKET} --delete

Why this check exists:

  • It models the publish step of a frontend release.
  • The --delete flag keeps the bucket aligned with the current build output, but it should be used carefully in production.

4. Invalidate the CloudFront cache after deployment

aws cloudfront create-invalidation \
  --distribution-id ${CLOUDFRONT_DIST_ID} \
  --paths "/*"

Why this check exists:

  • CloudFront may continue serving stale cached objects even after S3 has newer versions.
  • In production, targeted invalidations or versioned asset paths are usually better than always invalidating /*.

5. Test a Location Service search

aws location search-place-index-for-text \
  --index-name ${PLACE_INDEX_NAME} \
  --text "EV charging station" \
  --region ${AWS_REGION}

Why this check exists:

  • It isolates the map-data path from the rest of the frontend.
  • It helps prove whether a map issue is an edge problem, a Location Service configuration problem, or a frontend rendering problem.

Configuration Example

Private S3 origin policy for CloudFront OAC

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontServicePrincipalReadOnly",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::ev-prod-frontend/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E123ABC456DEF"
        }
      }
    }
  ]
}

What this configuration achieves:

  • The S3 bucket stays private.
  • Only the named CloudFront distribution can read objects.
  • Users must access the frontend through CloudFront, where WAF can inspect traffic.
  • The policy narrows origin access to a known distribution, which is more secure than allowing broad public reads.

Best practices for this layer

  • Prefer Route 53 alias records to CloudFront in production instead of CNAMEs when possible.
  • Use versioned frontend assets to reduce the need for large invalidations.
  • Attach AWS-managed WAF rules first, then add custom rules after baseline traffic is understood.
  • Consider CloudFront response headers policies for security headers such as Strict-Transport-Security and Content-Security-Policy.
  • Use separate Amazon Location resources for development and production to avoid accidental cross-environment changes.

What to Verify

  • DNS resolves to the expected CloudFront distribution.
  • The S3 bucket is not publicly readable.
  • A CloudFront invalidation is created after each deployment.
  • Map search works through Amazon Location Service.
  • WAF is associated with the CloudFront distribution in the target environment.
  • TLS certificates are valid and the frontend is served over HTTPS end-to-end.

Common Mistakes

  • Updating S3 content but forgetting to invalidate CloudFront.
  • Making the S3 bucket public instead of using OAC.
  • Pointing Route 53 to the wrong CloudFront domain.
  • Troubleshooting map failures in the backend when the real issue is Amazon Location configuration.
  • Treating CloudFront as only a cache while ignoring its role in edge security and origin protection.

Notes / Assumptions

  • The diagram shows Amazon Location Service as a frontend integration path, but it does not expose the exact map style or place index names.
  • Route 53 record type may be an alias rather than a CNAME in a real production setup. The example above uses a CNAME for clarity.
  • The diagram does not show API Gateway or Lambda at the edge. Dynamic requests move to the ALB-based application layer.
  • If geospatial access needs stronger client isolation, signed requests or a backend mediation layer may eventually be required.

Hands-on Private Application Routing and Compute

Overview

  • This section explains how dynamic requests move from the edge into private compute.
  • It focuses on the VPC, public and private subnets, Internet Gateway, NAT Gateway, ALB, EC2 instances, and Auto Scaling.
  • It is the primary section for troubleshooting 5xx errors, unhealthy targets, or outbound connectivity issues.
  • This subsection defines the core execution path for business logic, authentication, charging workflows, and other synchronous application behavior.

What this step does:

  • It receives public API traffic on the ALB.
  • It routes requests to healthy EC2 instances inside private subnets.
  • It provides controlled outbound access for those instances without giving them public IP addresses.

Why it exists:

  • The application needs a stable public API entry point but should not expose individual compute nodes.
  • The backend needs horizontal scaling and self-healing.
  • Private subnets lower the attack surface while NAT preserves outbound functionality.

What problem it solves:

  • Direct internet exposure of application servers
  • Lack of controlled traffic distribution across instances
  • Inability to replace failed nodes without downtime
  • Poor separation between inbound public access and outbound dependency access

AWS Services in This Section

Amazon VPC

  • Defines the network boundary for the entire system.
  • Hosts all subnets, gateways, and compute resources.
  • Makes it possible to separate public routing, private compute, and private data into controlled segments.

Internet Gateway

  • Connects the VPC to the public internet.
  • Enables inbound and outbound traffic based on route table configuration.
  • Supports public-facing resources such as the ALB and the NAT Gateway.

NAT Gateway

  • Allows instances in private subnets to access the internet.
  • Does not allow direct inbound traffic from the internet.
  • Acts as a controlled egress point for patching, dependency download, and calls to public AWS service endpoints.

Application Load Balancer

  • Receives external HTTP/HTTPS requests.
  • Distributes traffic to backend EC2 instances.
  • Enforces health-aware routing and can host listener rules for path- or host-based traffic decisions.

Amazon EC2

  • Runs backend application workloads.
  • Processes business logic and returns responses.
  • Holds the application runtime, local process state, and agent integrations such as SSM or CloudWatch logging.

EC2 Auto Scaling

  • Automatically scales the number of instances based on load.
  • Ensures high availability across multiple Availability Zones.
  • Replaces unhealthy or terminated instances to keep the target fleet at the desired size.

Why This Design Uses ALB + EC2

  • The diagram shows long-running application servers, so Amazon EC2 is the active compute model.
  • The public API entry point is Application Load Balancer, not API Gateway.
  • No Lambda, ECS, or EKS components are shown, so this workshop does not assume serverless or container orchestration.
  • Private subnets indicate that compute should not be directly reachable from the internet.
  • This design fits workloads that maintain application processes, connection pools, or JVM-based runtime characteristics that benefit from persistent hosts.

Step-by-Step AWS Flow

  1. The frontend sends a dynamic request to the backend hostname. Service: Application Load Balancer. What happens: The ALB receives the request at the public boundary of the application tier. Why it matters: Only the ALB should be public for backend traffic.
  2. The ALB evaluates listener rules and target group health. Service: Application Load Balancer. What happens: Requests are routed only to registered healthy targets. Why it matters: Bad instances can be isolated without taking down the whole service.
  3. A healthy EC2 instance in a private subnet processes the request. Service: Amazon EC2. What happens: The application performs business logic and prepares the response. Why it matters: Compute remains private and controlled by security groups.
  4. Auto Scaling keeps the desired number of instances running. Service: EC2 Auto Scaling. What happens: Failed instances can be replaced and capacity can grow during traffic spikes. Why it matters: The platform can recover from instance failure and scale out.
  5. If the instance needs outbound access, it sends traffic through the NAT Gateway. Services: NAT Gateway and Internet Gateway. What happens: Private instances reach external endpoints without receiving public IPs. Why it matters: The backend can call AWS public service endpoints or download dependencies without becoming internet-facing.
  6. The response returns through the ALB to the client. Service: Application Load Balancer. What happens: The ALB completes the HTTP request lifecycle. Why it matters: Clients never communicate with EC2 instances directly.

Request and response behavior

  • Inbound request behavior: The ALB terminates the public request and forwards it to an eligible target over the configured application port.
  • Backend processing behavior: The EC2 instance performs application logic, may call RDS or supporting AWS APIs, and returns an HTTP response.
  • Health behavior: If the instance fails the configured health check, the ALB stops sending production traffic to it.
  • Scale behavior: Auto Scaling adjusts capacity independently from request routing, allowing the service to adapt under load.

Internal logic of this layer

  • Public subnets exist to host internet-facing endpoints.
  • Private subnets exist to host compute that should not be directly reachable from the internet.
  • ALB health checks and target groups form the contract between the edge-facing application boundary and the actual backend nodes.
  • NAT is a convenience and security compromise: it preserves egress while keeping instances private, but it also introduces cost and availability trade-offs.

  • Keep instances stateless wherever possible so scaling events do not interrupt user sessions.
  • Use launch templates rather than manual instance builds for repeatability.

  • Use separate security groups for ALB, EC2, and RDS to make trust boundaries explicit.

  • Private subnets use a NAT route for outbound traffic.
  • Backend responses work even though instances do not have public IPs.
  • Security-group rules allow only the minimum required traffic between layers.

Common Mistakes

  • Opening the EC2 security group to the internet instead of only to the ALB.
  • Using a health check path that depends on the database and marks healthy instances as failed during database events.
  • Forgetting that private instances still need outbound connectivity through NAT.
  • Assuming API Gateway or Lambda is involved when the real entry point is the ALB.
  • Treating the NAT Gateway as optional even though package updates, AWS API calls, or third-party integrations may depend on it.

Notes / Assumptions

  • The diagram shows one NAT Gateway. In production, you may want one per Availability Zone for stronger outbound resilience.
  • Security groups and NACLs are not drawn, but they are critical to making this routing model work.
  • This workshop documents EC2-based deployment because ECS and EKS are not shown in the architecture.
  • If the product later exposes partner-facing APIs, API Gateway may be added in front of dedicated endpoints without replacing the current ALB path.

Relational Data Resilience and Network Isolation

Overview

  • This section describes how the platform stores persistent application data while keeping the database tier isolated from public access.
  • Its purpose is to provide durable relational storage and support availability in case of database or infrastructure failure.
  • It fits behind the application layer as the system of record for the workload.
  • In practice, this subsection defines the persistence contract for bookings, charging sessions, payments, user records, and operational reporting queries that still belong to the main transaction path.

What this step does:

  • It stores application data in a managed relational database.
  • It protects the data tier from direct public exposure.
  • It provides a resilient primary and standby posture for continuity.

Why it exists:

  • The EV platform needs strong transactional behavior rather than eventually consistent object storage.
  • Database availability is central to the user experience because booking, charging, and payment workflows all depend on consistent writes.
  • Data is usually the highest-value asset in the system and deserves the strongest isolation boundary.

What problem it solves:

  • Loss of transactional integrity
  • Public exposure of the database
  • Hardcoded host dependency on a single database node
  • Weak recovery posture during infrastructure or database failure

Components

Amazon RDS

  • Managed relational database service for the application data tier.
  • Handles provisioning, backups, patching, and failover.
  • Reduces operational overhead compared with self-managed database servers on EC2.

Primary Database Instance

  • Handles the main transactional workload.
  • Processes read and write operations from the application.
  • Represents the active endpoint for business-critical data changes.

Standby Database Instance

  • Maintains a synchronized replica of the primary instance.
  • Automatically takes over during failover scenarios.
  • Exists to improve resilience rather than to serve direct user traffic.

No Public Access Boundary

  • Ensures the database is not publicly accessible.
  • Access is restricted to internal resources within the VPC.
  • Reduces attack surface and prevents accidental exposure through broad CIDR rules.

EC2 to RDS Connectivity

  • Application servers connect to the database over private networking.
  • Typically uses security groups and internal DNS endpoints.
  • Keeps the application-to-database contract stable even if the underlying host changes during maintenance or failover.

Flow Description

  1. EC2 instances send database reads and writes to the primary RDS instance. What happens: The application opens a connection pool to the managed RDS endpoint and performs transactional operations. Why it matters: The application should depend on the stable endpoint, not a specific node IP.
  2. The RDS service maintains synchronization between the primary and standby instances. What happens: State changes are replicated according to the availability mode configured for the database. Why it matters: This is the basis for controlled failover and better durability.
  3. If the primary database becomes unavailable, the standby instance likely supports failover continuity. What happens: RDS shifts the primary role so the application can reconnect through the same logical endpoint. Why it matters: Failover time still matters, but application configuration changes should not be necessary.
  4. Database access remains internal to the application environment and is not exposed directly to internet users. What happens: Only allowed VPC-internal paths can reach the database port. Why it matters: The database remains protected even if a public endpoint elsewhere in the architecture is under attack.

Request and response behavior

  • Input request: The application submits SQL operations as part of synchronous business workflows.
  • Processing: The primary node handles reads and writes, while the standby remains in a resilience role.
  • Output response: The application receives query results, success acknowledgments, or database exceptions that it must handle cleanly.
  • Failure behavior: During failover or maintenance, connection retries and timeout handling become part of the application contract.

Key Logic / Rules

  • The primary database is the main endpoint for normal application transactions.
  • The standby database improves resilience but is not shown as a user-facing endpoint.
  • Network isolation is a core rule of the design, as the diagram explicitly marks the data tier with no public access.
  • Keeping the data tier private reduces attack surface and limits unintended connectivity paths.
  • Application-layer control is required for all database interactions.
  • Secrets should provide connection details so credentials and hostnames are not hardcoded in the application artifact.
  • If future read-heavy patterns emerge, read replicas or analytics-specific stores should be evaluated instead of overloading the primary path.

  --region ${AWS_REGION}

2. Display status, Multi-AZ flag, and endpoint

aws rds describe-db-instances \
  --db-instance-identifier ${DB_INSTANCE_ID} \
  --region ${AWS_REGION} \
  --query 'DBInstances[0].[DBInstanceStatus,MultiAZ,Endpoint.Address,Endpoint.Port]' \
  --output table

Use the failover command only when:

  • Multi-AZ or an equivalent standby mode is actually enabled,
  • the environment owner has approved the test,
  • the application team is ready to observe reconnect behavior.

Operational Access and Secret Governance

Overview

  • This section explains how administrators and application instances receive controlled access to the environment and sensitive configuration.
  • Its purpose is to reduce direct exposure, centralize secret handling, and enforce machine permissions through AWS-native controls.
  • It fits as the security and operations control plane alongside the runtime environment.
  • In operational terms, this is the subsection that explains how people and workloads gain trust without falling back to hardcoded credentials or public SSH.

What this step does:

  • It provides operator access to private instances through Systems Manager.
  • It gives EC2 instances machine identity through an IAM instance profile.
  • It externalizes sensitive values into Secrets Manager and protects them with KMS.

Why it exists:

  • Production systems should not distribute long-lived AWS keys or SSH keys to application hosts unnecessarily.
  • Operator access should be auditable and revocable.
  • Secrets should be centralized, encrypted, and rotated independently from application code releases.

What problem it solves:

  • Public administration paths to private servers
  • Hardcoded application credentials
  • Weak machine identity and permission sprawl
  • Inconsistent control over who can decrypt or use sensitive material

Components

AWS Systems Manager (SSM)

  • Provides controlled access for operators to connect to private EC2 instances.
  • Eliminates the need for SSH keys or public access.
  • Can support interactive sessions, command execution, and standardized runbooks.

IAM Instance Profile

  • Grants machine-level identity to EC2 instances.
  • Allows applications to securely access AWS services without hardcoded credentials.
  • Supplies temporary credentials that are rotated automatically by AWS.

AWS Secrets Manager

  • Stores sensitive data such as database connection strings, API keys, and credentials.
  • Provides secure retrieval of secrets at runtime.
  • Decouples secret lifecycle from application deployment lifecycle.

AWS KMS

  • Encrypts and protects secrets managed by the system.
  • Controls access to encryption keys via IAM policies.
  • Makes encryption governance a managed service concern rather than a custom application responsibility.

Amazon EC2

  • Consumes IAM roles and retrieves secrets during application runtime.
  • Acts as the execution environment for secure service interaction.
  • Uses machine identity to reach CloudWatch, SES, Secrets Manager, and other AWS APIs.

Flow Description

  1. An administrator reaches the environment through AWS Systems Manager rather than a direct internet-facing login path. What happens: The operator opens a brokered session to a private instance by targeting it through SSM. Why it matters: The instance can remain private while still being manageable.
  2. EC2 instances assume their IAM instance profile automatically at runtime. What happens: Temporary AWS credentials are provided through the instance metadata service. Why it matters: The workload can authenticate safely without embedding keys in files.
  3. The application reads secrets from AWS Secrets Manager when configuration or credentials are required. What happens: The runtime fetches named secret material on demand or at startup. Why it matters: Secret rotation and environment separation become much more manageable.
  4. AWS KMS likely protects the secret material or encryption operations associated with those secrets. What happens: Authorized callers can decrypt protected values while unauthorized callers are denied. Why it matters: Encryption is enforced at the service boundary and can be audited through AWS controls.
  5. The same IAM role also allows the instance to write logs and send email, as shown in the diagram. What happens: One machine identity covers multiple runtime integrations, but each permission should still remain tightly scoped. Why it matters: Centralized machine identity reduces credential sprawl while keeping service access explicit.

Request and response behavior

  • Operator request: SSM receives an access request and establishes a managed session if the target instance is registered and authorized.
  • Application request: The workload calls Secrets Manager to retrieve named configuration or credentials.
  • Crypto response: KMS authorizes or denies decryption depending on IAM and key policy alignment.
  • Service response: The instance either receives the required secret and credentials, or the failure is surfaced as a policy or configuration problem.

Key Logic / Rules

  • Administrative access is brokered through SSM, which avoids the need for public SSH or bastion exposure in the diagram.
  • Secret values are externalized into a managed service instead of being embedded directly on application servers.
  • IAM roles enforce least-privilege access for machine identities.
  • Encryption is treated as a managed capability rather than an application-owned key store.
  • The instance profile shown in the diagram is scoped to reading secrets, writing logs, and sending email.
  • IAM resource scope should be narrower in production than the simplified examples often shown in workshops.

AWS CLI Walkthrough

1. Start an SSM session to a private instance

aws ssm start-session \
  --target ${INSTANCE_ID} \
  --region ${AWS_REGION}

2. Verify the attached IAM instance profile

aws ec2 describe-instances \
  --instance-ids ${INSTANCE_ID} \
  --region ${AWS_REGION} \
  --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' \
  --output text

3. Read a secret value

aws secretsmanager get-secret-value \
  --secret-id ${DB_SECRET_ID} \
  --region ${AWS_REGION}

4. List KMS aliases

aws kms list-aliases --region ${AWS_REGION}

How to interpret these commands:

  • A successful SSM session proves both management-plane reachability and correct instance registration.
  • A missing instance profile ARN usually explains why secret retrieval, log publication, or SES calls fail later.
  • Secret retrieval failures commonly point to IAM permissions, wrong secret IDs, or KMS policy gaps.
  • KMS alias discovery helps verify whether the expected encryption boundary actually exists in the environment.

Best practices for this layer

  • Replace wildcard resource scope with secret-specific and key-specific ARNs in production.
  • Enable Session Manager logging to CloudWatch or S3 for auditability.
  • Separate operator roles from machine roles so human and workload permissions do not blur together.
  • Use secret naming conventions such as service/environment/purpose.
  • Consider automatic secret rotation for database credentials where the application can tolerate it.

What to Verify

  • The target EC2 instance is registered in Systems Manager.
  • The instance has an IAM instance profile attached.
  • The application can read the exact secret names it requires.
  • KMS permissions are granted only to the principals that truly need decryption.
  • Operators use SSM instead of opening inbound SSH from the internet.

Common Mistakes

  • Attaching a role to EC2 but forgetting secretsmanager:GetSecretValue.
  • Granting Secrets Manager access but missing kms:Decrypt when a customer-managed key is used.
  • Assuming end-user identity is handled here when the diagram only shows operator and machine access.
  • Returning to hardcoded credentials because secret naming is inconsistent.
  • Leaving broad IAM permissions in place after the workshop stage is complete.

Notes / Assumptions

  • The exact SSM features in use, such as Session Manager or Run Command, are not labeled explicitly.
  • Secret rotation and KMS key rotation policies are not shown.
  • No separate identity provider or fine-grained admin workflow engine appears in the diagram, so the documentation stays focused on the visible AWS controls.
  • In a hardened environment, private VPC endpoints for SSM, Secrets Manager, and CloudWatch may reduce dependence on NAT for control-plane traffic.

Hands-on Monitoring, Audit, and Email Delivery

Overview

  • This section explains how operators observe the running platform, audit AWS activity, and verify outbound email delivery.
  • It focuses on Amazon CloudWatch, AWS CloudTrail, Amazon SES, and Route 53 email DNS records.
  • It is the right section when the team needs to inspect logs, trace operator actions, or validate the email path used by the application.
  • In production terms, this subsection closes the loop between application behavior, operator visibility, and customer-facing communications.

What this step does:

  • It captures operational signals from the running environment.
  • It records control-plane actions for security and compliance visibility.
  • It enables the application to send email through a managed outbound service.

Why it exists:

  • Without logs and metrics, teams diagnose incidents too slowly.
  • Without audit trails, teams cannot reliably answer who changed what and when.
  • Without a managed outbound email service, transactional messaging becomes more fragile and operationally expensive.

What problem it solves:

  • Low visibility during incidents
  • Missing audit evidence for operator actions
  • Self-managed email complexity
  • Weak confidence in notification delivery

Components

Amazon CloudWatch

  • Stores logs and metrics from the running environment.
  • Enables monitoring, alerting, and troubleshooting of application behavior.
  • Serves as the main operational telemetry plane for the workload.

AWS CloudTrail

  • Records AWS API activity across the account.
  • Supports auditing, security analysis, and compliance tracking.
  • Makes infrastructure and operator actions traceable long after the immediate incident is over.

Amazon SES

  • Sends application emails such as notifications or verification messages.
  • Requires verified domain and proper DNS configuration.
  • Offloads SMTP infrastructure and reputation management to a managed service.

Route 53 DNS Records (SES Support)

  • Provides DNS records for domain verification and email routing.
  • Includes DKIM, SPF, and MX records for SES integration.
  • Acts as the infrastructure prerequisite for trusted sender identity.

IAM-backed EC2 Role

  • Grants EC2 permission to:
    • Publish logs to CloudWatch
    • Send emails via SES
  • Eliminates the need for hardcoded credentials.
  • Keeps machine identity consistent with the rest of the workshop’s security model.

Why This Design Uses Direct AWS Integrations

  • The diagram shows CloudWatch and CloudTrail directly, so observability is centered on AWS-native tooling.
  • Email is sent directly through Amazon SES rather than via a separate messaging or event bus service.
  • EventBridge, SQS, and SNS are not shown in the architecture, so this workshop does not assume an event-driven notification pipeline.
  • This direct pattern keeps the architecture easy to understand, but it also means request-path side effects may remain tightly coupled until asynchronous services are introduced later.

Step-by-Step AWS Flow

  1. The application emits logs and runtime metrics. Service: Amazon CloudWatch. What happens: Application and platform signals are written to log groups and CloudWatch metrics. Why it matters: Logs and metrics are the first place to look when user-facing errors appear.
  2. Operators inspect the live logs. Service: Amazon CloudWatch Logs. What happens: The team tails or queries log streams to find failures and timing issues. Why it matters: Fast diagnosis depends on finding the correct log group quickly.
  3. AWS control-plane activity is recorded. Service: AWS CloudTrail. What happens: Actions such as starting an SSM session or changing infrastructure are stored as audit events. Why it matters: Operators can tell who changed what and when.
  4. The application sends email through SES. Service: Amazon SES. What happens: The EC2 workload calls SES APIs with its IAM role. Why it matters: Notifications, OTP, or operational messages can leave the platform without self-managed SMTP servers.
  5. SES relies on DNS records in Route 53. Services: Amazon SES and Route 53. What happens: Domain verification, DKIM, and mail routing records are published in the hosted zone. Why it matters: Without valid DNS records, SES identity verification or deliverability will fail.

Request and response behavior

  • Telemetry request: The application sends structured log lines or metrics to CloudWatch.
  • Audit response: CloudTrail receives service activity as AWS-generated audit events.
  • Email request: The application submits an outbound email payload to SES and receives success or failure status.
  • DNS verification behavior: SES identity status changes only after the required Route 53 records are present and propagated.

Operational response pattern

  • CloudWatch helps explain what the application did.
  • CloudTrail helps explain what operators or AWS services changed.
  • SES and Route 53 together help explain whether the system can communicate with users outside the platform boundary.

aws route53 change-resource-record-sets \
  --hosted-zone-id ${HOSTED_ZONE_ID} \
  --change-batch file://ses-identity-records.json

Best practices for this layer

  • Use structured logs so application events can be filtered and queried consistently.
  • Define retention periods separately for application logs, security logs, and audit logs.
  • Treat CloudTrail as mandatory governance data, not optional debugging data.
  • Handle SES bounces and complaints if the workload becomes customer-facing at scale.
  • Use alarms that point to an actual operational response process rather than creating alert noise.

What to Verify

  • Application logs arrive in the expected CloudWatch log group.
  • CloudTrail captures administrator and infrastructure actions.
  • SES identities are verified before the application attempts delivery.
  • Route 53 contains the TXT, CNAME, and MX records required by your SES setup.
  • CloudWatch alarms exist for the signals your team cares about.
  • The EC2 instance role includes only the permissions required for telemetry and messaging.

Common Mistakes

  • Sending email before verifying the SES identity.
  • Looking only at application logs and ignoring CloudTrail when the issue is an operator action.
  • Creating alarms without deciding who will review them.
  • Expecting EventBridge or SQS-driven notification workflows when the architecture only shows direct SES integration.
  • Treating email delivery as purely an application concern and forgetting the DNS dependency.

Notes / Assumptions

  • The diagram does not show dashboards, dashboards-as-code, tracing, or alarm actions.
  • The alarm example intentionally avoids SNS wiring because SNS is not part of the visible architecture.
  • Email use cases such as OTP or notifications are inferred from the presence of SES, but the exact templates are not shown in the diagram.
  • If notification volume or retry complexity grows, EventBridge, SQS, or SNS would be natural additions.

Architecture Baseline

The workshop describes the following AWS runtime baseline:

User -> Route 53 -> CloudFront/WAF -> S3
Frontend -> Amazon Location Service
Frontend -> ALB -> Private EC2 -> Amazon RDS
Private EC2 -> Secrets Manager/KMS -> CloudWatch -> SES
Admin -> Systems Manager -> Private EC2
AWS control-plane activity -> CloudTrail

Public and private boundary

  • Route 53, CloudFront, WAF, and the ALB define the public-facing entry path.
  • EC2, RDS, Secrets Manager, KMS, and most operational controls are part of the protected internal estate.
  • SES is public as a service endpoint, but it is used by a private workload through authenticated AWS APIs.

What this architectural step does

  • It shows the logical service graph of the platform.
  • It clarifies which services belong to the user request path and which belong to the supporting control plane.
  • It provides a practical basis for scaling, fault tolerance, and security discussions.

Request Lifecycle and Service Interaction Flow

Lifecycle 1: User enters the platform

  1. Route 53 resolves the domain.
  2. CloudFront and WAF process the request.
  3. Static frontend content is returned from cache or S3.

Lifecycle 2: User interacts with maps

  1. The frontend requests place or map data.
  2. Amazon Location Service responds directly to that integration.

Lifecycle 3: User performs a business action

  1. The frontend sends a dynamic request through the ALB.
  2. A healthy private EC2 instance processes the action.
  3. The app reaches RDS, CloudWatch, Secrets Manager, or SES as needed.

Lifecycle 4: Operator manages the system

  1. An operator connects through Systems Manager.
  2. Actions are later visible in CloudTrail.
  3. Runtime effects remain visible in CloudWatch.

Natural Evolution Paths

  • Introduce API Gateway only when API management needs exceed what ALB naturally provides.
  • Introduce Lambda for helper automations rather than replacing the current EC2 backend prematurely.
  • Introduce Step Functions if booking, charging, and settlement flows become multi-step and long-running.
  • Introduce EventBridge, SQS, or SNS if background processing needs looser coupling from the request path.
